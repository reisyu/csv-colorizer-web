# Geometry Attribute Colorizer 仕様書

作成日: 2026-07-09
対象バージョン: index.html(単一ファイル構成)

この文書は、機能の仕様に加えて「なぜその設計にしたか」という意図を記録するためのものです。
今後の機能追加・変更の際は、ここに書かれた意図と矛盾しないかを判断基準にしてください。

---

## 1. 目的と背景

### 1.1 解決したい課題

石垣(伝統的石積み)の調査ワークフローにおいて、各石の幾何属性(幅・高さ・面積など)を
計算し、属性に応じて色分け可視化する工程は、従来Grasshopper(有償のRhinoceros上で動作)で
行われていた。有償ソフトが前提であることが、調査関係者への普及の障壁になっていた。

**本アプリの目的: この属性計算・可視化の工程を、無償・インストール不要のブラウザ
アプリとして提供し、より多くの人が使えるようにすること。**

### 1.2 想定ワークフロー全体における位置づけ

1. 石垣をフォトグラメトリで3Dモデル化(OBJ + MTL + テクスチャPNG) — 外部ソフト
2. クラスタリングソフトで各石の輪郭を抽出(DXF、閉じた3Dポリライン) — 外部ソフト
3. **本アプリ: DXFを読み込み、属性を自動計算し、色分け表示・図版出力・データ書き出し**
4. 必要に応じてCSV/GLBで書き出し、他ツール(Blender等)で分析・再利用

ステップ1・2のWebアプリ化は検討の結果見送った(フォトグラメトリは計算量的に非現実的、
輪郭抽出は既存ソフトの品質再現コストが高い)。ただしステップ1の成果物(OBJ)を
「背景モデル」として重ね表示する機能は搭載した。

### 1.3 開発経緯(意図の変遷)

- 当初は「GLB + 外部CSVを読み込んで色分けする」ビューアーとして出発
  (アプリ名も当初は CSV Attribute Colorizer だった)
- その後、Grasshopperで行っていた属性計算そのもの(平面化・最小外接矩形・
  幅高さ判定・充填率)をアプリ内に実装。**CSVを外部で用意する必要がなくなり、
  DXFを放り込むだけで完結する形へ発展した**
- これに伴い、実態に合わせてアプリ名を Geometry Attribute Colorizer に変更した
- 同機能のBlenderアドオン版も並行開発したが、Webアプリが大きく先行したため
  アドオン版は凍結。Blenderが必要な場面は「ID付きGLB書き出し→Blenderで読む」で
  接続する方針とした

## 2. 設計原則

以下は個別機能に優先する全体方針である。

| 原則 | 内容と意図 |
|---|---|
| **単一HTMLファイル** | index.html 1つで完結。配布・GitHub Pages公開・オフライン利用が容易。外部依存はCDNのThree.jsとPapaParseのみ。GLB/OBJ/DXFのパーサーは全て自前実装(CDNに存在しないローダーへの依存で動かなくなった教訓による) |
| **ブラウザ内完結** | 読み込んだファイルは一切サーバーへ送信しない。調査データは公開前の機密であることが多く、プライバシーはツールの信頼性そのもの |
| **iPad対応** | 現場での利用を想定。accept属性を使わない(iOSでグレーアウトする)、タッチ操作(ピンチズーム・2本指パン)、inputmode="decimal"、アコーディオンUIで縦長パネルを圧縮 |
| **大規模データ耐性** | 実データは2,500〜10,000オブジェクト規模。重い処理はチャンク分割(200個単位)+プログレスバーでUIを固めない |
| **用語は「オブジェクト」で統一** | 石垣特化ではなく汎用ツールとしての表現を採る。「石」「輪郭」はサンプルの説明など内容記述に限る |
| **検証してから出す** | 幾何計算は既知の入力での検証(Pythonでの再現テスト)を経てから組み込む |

## 3. 座標系と原点シフト(最重要の設計判断)

### 3.1 前提

- 入力データ(DXF/OBJ)は **Z軸が鉛直(上)、XY平面が水平** のZ-up座標系
- 座標値は平面直角座標系などの実座標(例: X≈-43,000、Y≈-145,000)であり、
  原点から数万〜数十万単位離れている

### 3.2 問題と対策

GPU(WebGL)の頂点座標はfloat32(有効約7桁)であり、大きな座標をそのまま使うと
精度が崩壊する(頂点のガタつき、Zファイティング)。そのため:

- **読み込み時に全体のバウンディングボックス中心を原点へシフト**する
- シフト量は `worldOffset` というグローバルな共有値として保持する
- **先に読み込んだデータ(通常DXF)がシフト量を決め、後から読むデータ(OBJ背景)は
  同じ値を再利用する**。これにより実座標系のまま両者がぴったり重なる
  → 運用上「DXFを先に読み込む」ことを推奨(READMEにも明記)
- Three.js内部はY-upのため、`(x, y, z) → (x-ox, z-oz, -(y-oy))` で変換する
- 属性のXYZ座標は**シフト前の元の値**で出力する(GIS等との照合のため)

## 4. 入力データ仕様

### 4.1 DXF(主入力)

- 対応形式: **POLYLINE / VERTEX エンティティ、フラグ70=9(閉じた3Dポリライン)**
- 1ポリライン = 1オブジェクト(石)
- **レイヤー名の末尾の数字が読み込み時のIDになる**(例: `Layer123` → `123`)
- 輪郭は完全な平面でなくてよい(実データは平面から平均28mm・最大108mm程度ズレる。
  最適フィット平面に投影して計算する)
- パーサーは自前実装。上記形式のみ対応(LWPOLYLINE等は非対応)。
  **クラスタリングソフトの開発元にはフォーマットを変えないよう依頼する方針**

### 4.2 GLB(glTF 2.0 バイナリ)

- 自前パーサーで読み込み(ヘッダー/JSONチャンク/BINチャンク解析)
- ノード名がオブジェクト名(=CSVマッチング用ID)になる
- .gltf+.bin分離形式は非対応(エラーメッセージで案内)
- GLBモデルは三角形メッシュのため、属性の自動計算・SVG書き出しの対象外。
  外部CSVとの組み合わせで色分けする用途

### 4.3 CSV(外部属性)

- 1行目ヘッダー、ID列 + 任意の属性列。数値/テキストは自動判定
- ID正規化: `"1"`, `"1.0"`, `" 1 "` の表記ゆれを吸収して照合する
- 欠損値(空欄)は安全にスキップ(実データに欠損があった経験から)

### 4.4 OBJ + MTL + テクスチャ(背景モデル)

- フォトグラメトリモデルの重ね表示専用。属性計算・クリック判定・書き出しの対象外
- ブラウザの制約上、OBJ選択時に隣のMTL/PNGを自動で読むことは不可能。
  **「複数ファイル同時選択」または「フォルダごと選択(webkitdirectory)」**で対応
- **テクスチャは読み込み時に最大2048pxへ自動縮小**(意図: 実データは25MB PNG×15枚で、
  原寸ではGPUメモリを数GB消費し破綻するため。背景用途では2048pxで十分)
- マテリアルはMeshBasicMaterial(意図: フォトグラメトリのテクスチャは陰影焼き込み済み
  なので照明計算は不要。軽く、実物の見た目に忠実)
- 描画は両面(DoubleSide)。輪郭表示も同様(意図: Rhino系データは法線の向きが
  まちまちで、片面描画だと面が消える)

## 5. 自動計算される属性

DXF読み込み時、各オブジェクトについて以下を計算しCSV化する。
列名は英語(他ツールとの互換性)、UI表示は「Width(幅)」のように日本語併記。

| 列名 | 日本語名 | 定義 | 実装上の要点 |
|---|---|---|---|
| Width | 幅 | 最小外接矩形の2辺のうち水平に近い方 | 矩形の2軸を3Dに戻し、Z成分(鉛直成分)が小さい方の軸を幅とする |
| Height | 高さ | もう一方の辺 | 同上 |
| Area | 面積 | フィット平面に投影した輪郭の面積 | 靴ひも公式 |
| FillRate | 矩形充填率 | Area ÷ 外接矩形の面積 | 0〜1。1に近いほど矩形に近い形。全データで0〜1に収まることが計算の正しさの検証にもなる |
| Tilt | 傾斜角 | 面の法線と鉛直(+Z)のなす角(度) | 0°=水平面、90°=垂直面。法線の符号に依存しないよう0〜90°に正規化 |
| RectAngle | 矩形回転角 | 外接矩形の水平辺が「面内の水平基準」からなす角(度) | 面内水平基準=法線×鉛直(走向方向)。反時計回り(右上がり)が正、±45°の範囲。**面内で測る**(3D空間の絶対角ではない)ことがユーザー要件 |
| Flatness | 平面性 | フィット平面からの最大ズレ距離 | 大きいほど平面近似の信頼性が低い。品質フィルタ用 |
| SelfIntersect | 自己交差 | 輪郭の辺同士が交差していれば1 | 隣接しない全辺ペアの線分交差判定。異常輪郭の検出用 |
| X, Y, Z | 重心座標 | 輪郭重心の**シフト前**の座標 | GIS照合用。Zで色分けすると標高グラデーションになる |
| SrcID | 元ID | 読み込み時のレイヤー名由来ID | 自動採番(§7)で振り直される前の値。クラスタリングソフト側との照合用トレーサビリティ |

### 5.1 計算アルゴリズムと選定理由

1. **法線: Newell法** — 閉じたほぼ平面のポリゴンに対して安定。PCAより軽量で、
   輪郭(順序付き頂点列)に適する
2. **平面化** — 重心を通るフィット平面に頂点を投影して2D化
3. **最小外接矩形: 回転キャリパー法** — 「最小矩形の一辺は凸包のいずれかの辺と
   重なる」という定理により、凸包の辺の数だけ試せば**厳密解**が得られる。
   対象が「完全に平面」というユーザー確認を得て、3D版(O'Rourke法)ではなく
   2D版で十分と判断した
4. **凸包: Andrew's monotone chain** — O(n log n)、実装が簡潔

### 5.2 既知の限界(意図的に保留したもの)

- **RectAngleの正方形問題**: Width≒Heightのオブジェクトは水平辺の判定が僅差で
  反転し、値が±45°付近で不安定になる。対策(フラグ付与など)は検討したが
  **ユーザー判断で保留中**
- テキスト属性のカテゴリ色は8色パレットの使い回し(9種以上で重複)

## 6. 色分け・可視化

### 6.1 色分け

- 数値属性: グラデーション(赤→青 / 赤→緑 / 黄→青 / グレースケール / カスタム)。
  既定の向きは「最小値=赤」(ユーザー指定による)
- カスタムは開始色・終了色に加え**中間色**を挟んだ3色補間に対応(例: 赤→白→青。既定色は開始=赤、終了=青)
- **反転**チェックで全配色の向きを逆転できる
- **範囲の手動指定**: 最小・最大を固定できる(意図: 外れ値で色が潰れる対策、
  複数モデル間で色の基準を揃える用途。RectAngleを±45固定+赤白青にすると
  傾き方向が対称に可視化できる)
- **設定変更は即時反映**(一度実行した後)。カラーピッカーは150msデバウンス。
  再適用時はマテリアルを作り直さず色だけ更新する(1万個規模への配慮)

### 6.2 表示技術

- オブジェクトごとにMeshStandardMaterialを持たせ、色分け時は色のみ更新
- 凡例(左下): 属性名+件数、**ヒストグラム(24ビン、各ビンを実際の
  グラデーション色で着色)**、グラデーションバー、最小・最大値。
  ヒストグラムをバーの真上に置くのは「どの色のオブジェクトが何個あるか」を
  直接読めるようにする意図
- ラベル: 各オブジェクトの重心位置にHTML要素を毎フレーム投影配置。
  複数属性の同時表示、文字サイズ調整、**距離カリング**(遠いラベルを自動非表示。
  ラベル過密と描画負荷の両方への対策)
- クリック(タップ)で属性インスペクタ表示(右上)

### 6.3 フィルタ

- **任意の数値属性 × 範囲(最小/最大)** の条件を複数追加できる(AND)。
  当初はArea/Flatness固定だったが、「Tiltでも絞りたい、他の属性でも」という
  要望を受けて汎用化した
- 自己交差オブジェクトの除外チェック
- 除外の表現: **半透明(既定)/ 非表示**。半透明を既定にしたのは
  「何が除外されたかを確認しながらしきい値を調整できる」ため
- 除外オブジェクトはラベル・SVG書き出しからも除かれる

## 7. ID自動採番

### 7.1 仕様

- DXF/サンプル読み込み時、**立面図ビュー(§8の投影)で左下のオブジェクトから
  順に 1, 2, 3... と自動採番**する(ボタン操作不要)
- 採番順: 下の段から、段内は左から右
- 段の判定: 投影後の重心高さでソートし、「重心高さの差がオブジェクト高さの
  中央値の60%以内」なら同じ段とみなすバンド分割(模擬データ6段×65石で
  正しく分離することを検証済み)
- 元のID(レイヤー名由来)は **SrcID属性として必ず保持**する。
  意図: 自動で振り直すことによるトレーサビリティ喪失の防止。CSVがそのまま
  新旧IDの対応表になる
- 手動の「IDを振り直す」ボタンも残す(フィルタ後の再採番など)
- GLB読み込みは自動採番の対象外(名前がCSVマッチングに使われるため)

## 8. 立面図SVG書き出し

### 8.1 投影の定義

- **全オブジェクトの面法線を平均し、水平化した方向から見た正射投影**
  (=壁の正面・水平視線からの正式な立面図。3Dビューのカメラ位置に依存しない)
- 検証: 水平方向の寸法は実寸のまま、高さは壁の後傾分だけ見かけ通りに縮む、
  という立面図の性質を満たすことを数値検証済み
- **表裏判定**: 「石垣の外向き法線は上向き(Z+)成分を持つ」という性質を利用し、
  各法線を上向き成分が正になるよう統一してから平均する。これにより最初の
  オブジェクトの頂点巻き方向に依存せず、常に表(外面)から見た図になる。
  法線がほぼ水平のオブジェクトのみ参照法線への揃えにフォールバック

### 8.2 出力内容

- 各オブジェクトの輪郭ポリゴン(現在の色分け色、細い輪郭線、奥→手前の描画順)
- 凡例(属性名+件数、ヒストグラム、グラデーションバー、最小・最大値)
- **スケールバー**(右下。モデル幅の約1/6に近い切りの良い長さを自動選択、単位m)
- ラベル(表示ONのとき、選択中の内容をテキストとして埋め込み)
- フィルタで除外したオブジェクトは出力しない
- 白背景・sans-serif。報告書・論文用にIllustrator等で後編集できる完全ベクター

### 8.3 対象の制限

- **輪郭モデル(DXF/サンプル)のみ**。ジオメトリの頂点順序=輪郭であることを
  利用しているため(`mesh.userData.isContour`フラグで判定)。
  GLB(三角形メッシュ)と背景OBJは対象外

## 9. その他の書き出し

- **CSV**: 現在の属性テーブル(自動計算 or 外部CSV)をBOM付きUTF-8で保存
- **ID付きGLB**: 表示中のメッシュを、名前(ID)を保持した標準GLBとして自前で
  組み立てて書き出す。往復テスト(書き出し→自前パーサーで読み戻し)で名前・
  座標の保持を検証済み。**DXF→GLB変換**としても機能し、凍結したBlenderアドオン
  ワークフローへの接続点でもある

## 10. UI構成

サイドパネルは6ステップのアコーディオン(ヘッダークリックで開閉)。
初期展開は「1. モデル」「3. 色分け」のみ(意図: iPadでの縦長パネル対策と、
DXFワークフローでは2が不要なため)。

1. **モデルを読み込む** — GLB/DXF選択、フォトグラメトリ背景(ファイル複数選択 or
   フォルダ選択)、背景の不透明度・表示・削除
2. **属性データを読み込む** — CSV選択、ID列名、接頭辞、ID振り直しボタン。
   DXF読み込み時は「✓ 自動計算した属性を使用中」と表示され、飛ばしてよいことを示す
3. **色分けする** — 属性選択(選ぶと定義の説明を表示)、配色、反転、範囲手動指定
4. **フィルタで絞り込む** — 除外条件(複数)、自己交差除外、表示方法
5. **ラベルを表示する** — 表示内容(複数選択)、文字サイズ、距離カリング
6. **書き出す** — CSV / ID付きGLB / 立面図SVG

その他:
- ステータス表示はパネル最上部(結果・警告をスクロールなしで見せる)
- サンプル表示中はビューポート左上にバッジ表示(ボタン文言を状態表示に
  使わない。操作=ボタン、状態=バッジという役割分担)
- 操作: 左ドラッグ回転 / 右ドラッグ・Shift+ドラッグ・2本指ドラッグでパン /
  ホイール・ピンチでズーム / クリックで属性表示

## 11. 内蔵サンプル

- 起動時に石垣風のサンプル(約69オブジェクト・6段・後傾18°・目地8mm)を自動生成し、
  色分けまで済んだ状態で表示する(意図: 初見の人が開いた瞬間に何ができるアプリか
  伝わるようにする)
- 生成は乱数シード固定(誰が開いても同じ)。下段ほど大きい石、千鳥の目地、
  石ごとの面内回転±12°・面外傾き±6°・微小な面外ノイズを持たせ、
  **全属性(Tilt・RectAngle含む)に意味のある分布が出る**ようにしている
- 属性計算はDXFと同じ経路(computeContourAttributes)を通す。サンプル自体が
  計算ロジックの動作デモを兼ねる
- 同一ロジックから生成した `sample_ishigaki.dxf` をリポジトリに同梱
  (実データの公開可否に依存しない配布用。実座標風オフセット付きで
  原点シフト機能のデモにもなる)

## 13. 既知の課題・今後の検討事項

- RectAngleの正方形問題(§5.2、保留中)
- カテゴリ色8色上限
- 選択オブジェクトの3Dハイライト(インスペクタとの対応が分かりにくい)
